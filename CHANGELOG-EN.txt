[2026-06-09] - Guild channel delivery, custom house design editor, script verb completeness (Wave 76)
----------------------------------------

ENGINE - speech:
  - Guild/alliance channel chat is now delivered. The channel event carries the
    recipient (it previously invoked with no recipient, so nothing could be
    routed), the host subscribes it and sends 0xAE with talk type 0xD/0xE per
    member, the speaker gets the server echo, and alliance delivery requires a
    mutual (both-sides declared) alliance. Guild/alliance lines are no longer
    spatially broadcast to bystanders near the speaker.

ENGINE - custom housing:
  - The house design editor is implemented end to end. Incoming 0xD7 encoded
    commands now carry their payload (it was previously discarded after the
    subcommand) and are handled: Build, Delete, Stairs, Roof, RoofDelete, Level,
    Clear, Backup, Restore, Sync, Revert, Commit, Close. A per-character design
    session holds the working design; Commit persists it as DESIGN_n tags on the
    multi item (same format the script ADDITEM verb writes) and bumps
    DESIGN_REVISION.
  - Outgoing design packets: 0xD8 design stream (zlib-compressed mode-0 planes,
    byte layout verified against the client parser, large designs split across
    planes), 0xBF 0x1D revision notice, 0xBF 0x20 customization mode on/off.
    The 0xBF 0x1E design query is now a known subcommand answered from the
    committed design. The house sign gump gained a "Customize House" entry for
    owners/co-owners, and a committed design broadcasts the new revision to
    nearby clients so they re-query.
  - @HouseDesignCommit and @HouseDesignExit now fire (Commit/Close paths).

ENGINE - experience:
  - Experience/level runtime: ChangeExperience routes every EXP change (script
    writes included) through @ExpChange (ARGN1 adjustable, RETURN 1 cancels);
    crossing a level threshold fires @ExpLevelChange with the new level. Killing
    an NPC awards the victim's EXP value to the killer — characters without EXP
    grant nothing, so the system stays inert unless definitions/scripts assign
    experience. Level curve is configurable (LevelNextAt step, linear or
    doubling mode).

ENGINE - help menu:
  - The help-menu buttons do real work now: "stuck" teleports to a safe town
    spot and is refused while jailed or in combat; "page" opens a text-entry
    gump routed through the .PAGE command path (staff broadcast + a capped
    recent-page log); "page list" shows recent pages (staff see all, players
    their own).

ENGINE - client packets:
  - 0xA5 web link writer added, exposed as the WEBLINK script verb and a
    "Visit Web Page" button on the guild stone gump when the guild has a web
    URL set.
  - Ground-drop sound is item-aware: gold coins play the amount-scaled coin
    sounds (0x2E4..0x2E6) instead of the generic 0x42.
  - @UserVirtue (0xBF 0x2C) and @UserKRToolbar (0xBF 0x24) now fire.

SCRIPTING - verbs:
  - Item OPEN opens the container on the acting client with script authority;
    DCLICK/USE replay the real client double-click path (doors, potions,
    containers). Character DISMOUNT performs a full dismount through the mount
    engine (client path fires @Dismount and refreshes the rider; flag-clear
    fallback remains for headless runs).
  - Object TRIGGER fires the named trigger through the dispatcher's new by-name
    path, so custom trigger names defined only in EVENTS/TEVENTS work. FIX
    re-seats the object Z on the terrain surface. TAGLIST (character, region,
    room) dumps tags to the invoking console. Region/Room ALLCLIENTS delivers a
    sysmessage or runs a script function for every online client character in
    the area.

DIAGNOSTICS:
  - Unimplemented SERV.* verbs are no longer silent: first use of each verb is
    warn-logged (debug afterwards) and GM-level callers get an in-game notice.
  - FILE.OPEN failures report their reason through a host-wired diagnostic sink
    instead of being swallowed; bare catch blocks in the recording loader,
    host IPC connect retry and stable-pet tag parser were narrowed to the
    expected exception types.

INTERNAL:
  - GameWorld now documents its threading contract (single-writer plain
    collections vs concurrent dirty/region caches; parallel stages must queue
    intents for the serial phase). Stub inventory updated; remaining gaps are
    marked as deliberate deferrals with reasons.

[2026-06-07] - Pet-command speech gating, skin hue preserved across death/resurrect (Wave 75)
----------------------------------------

ENGINE - speech:
  - The pet-command parser no longer treats every two-word line as "<pet name>
    <command>"; it only engages when the line actually starts with a pet command
    verb, so ordinary two-word speech no longer prints the "Sorry" pet-failure
    message when the speaker has no pet.

ENGINE - death / resurrect:
  - Skin hue is preserved across death. The living body's skin hue is captured into
    OSkin before the death spell-cleanup runs, and a dead character logging in keeps
    that hue instead of losing it when switching to the ghost hue. Fixes the
    character staying grey/default-hued after resurrect.

ENGINE - NPC definitions:
  - CHARDEF skill ranges (WRESTLING, TACTICS, MAGERY, etc.) are now parsed and
    rolled onto the character on spawn / applyDefName (CharDefHelper
    ApplyNpcDefinitionSkills, clamped 0..1200). Definition TAG defs are applied to
    the runtime character too (ApplyNpcDefinitionTags), so AI that reads BREATH.*,
    THROWOBJ, THROWDAM, THROWRANGE for dragon/breath/throw special attacks now sees
    the values from the definition (previously the AI read tags that were never set).

ENGINE - combat:
  - Hit-chance gained a stat-based fallback for NPCs whose weapon skill or Tactics
    is 0: the inferred value is clamp(max(Str,Dex)*10, 250..1000). It only applies
    to non-player attackers and weapon skills, so player accuracy is unchanged. This
    stops skill-less NPC definitions from missing almost every swing.

SCRIPTING - definition parsing:
  - Fixed bare Sphere hex parsing in ItemDef and CharDef: values like 0F5E / 013B
    are no longer mistaken for decimal, and the {min max} form is parsed for item
    damage/range. This also closes a path where an item definition could load with
    wrong metadata (e.g. a sword mis-typed so it emitted the bow/arrow shoot sound).

HOST / PANEL / BUILD:
  - Replaced the in-app panel update with a single update.cmd / update.ps1, run
    with the server stopped: git pull --ff-only -> full build.ps1 (panel + Server +
    Host) -> copy the build output into the run (deploy) folder -> relaunch
    SphereNet.Host. Running as a separate process it can replace SphereNet.Host.exe
    too, which the in-app updater could not. The source repo comes from sphere.ini
    AppUpdateRepoDir (or -RepoDir); the deploy folder defaults to the script's own
    folder (or -DeployDir) and the build-output==deploy case skips the copy.
    build.ps1 copies update.* into the build output so a deploy folder self-updates.
  - Removed the in-app UpdateService, POST /api/update/run, GET /api/update/status,
    the Settings "System Update" card and the AutoUpdate startup check.
    AppUpdateRepoDir is now read only by update.ps1; the AutoUpdate key is gone.

ENGINE - sound / visual / effects:
  - Added 0xC0 PacketEffectHued, 0xC7 PacketEffectParticle and 0x23
    PacketDragAnimation outgoing packets.
  - ObjBase.SOUND / EFFECT are no longer stubs: item- and char-based script
    commands now broadcast to nearby clients. Character SOUND / EFFECT route through
    the shared ObjBase path, and FACE now triggers a 0x77 facing broadcast.
  - Pickup and ground-drop emit the 0x23 drag animation, filtered to clients that
    support it (KR / Enhanced / SA).
  - Region change resends GlobalLight, a silent Season and Weather to the client.

ENGINE - combat & movement:
  - NPC walk steps now always face the step direction (surround step, Wander, flee
    and move-away paths previously moved without setting Direction, so the 0x77 move
    mismatched the tile delta and the client snapped the NPC). The surround step is
    also constrained to tiles adjacent to the NPC, so it can no longer jump two
    tiles in one move ("NPC teleport" during combat).
  - A player turning to face a combat target now also updates the attacker's own
    client (0x20 PlayerUpdate + walk-seq reset), mirroring Source-X
    CChar::UpdateMove -> addPlayerView/addPlayerUpdate, so in top-down view the
    avatar visibly faces the target instead of only nearby observers seeing it.

ENGINE - items & visuals:
  - Ground items carry a facing direction: 0x1A and 0xF3 world-item packets send it,
    the per-client known-item cache tracks it, and WorldSaver persists DIR. Dropping
    separate (non-stacking) items on one tile scatters their facing by the tile's
    existing item count instead of every drop landing on Direction=1.
  - A worn item previously known to a client as a ground item is no longer
    0x1D-deleted by the per-tick view delta (the 0x2E worn packet already re-homes
    it) - fixes worn items vanishing until a resync after a GM recolour/property edit.
  - GM x-verbs on item targets (.xcolor / .xid / ...) refresh the item visual for
    all observers; previously only COLOR (via OnVisualUpdate) and character targets
    refreshed.

ENGINE - persistence:
  - Periodic world auto-save: sphere.ini SavePeriod (minutes, 0 = off) now drives a
    save on the main loop. It was read into config but never used; with
    auto-save-on-shutdown disabled, the server never saved on its own.

CONFIG:
  - Notoriety name hues (ColorNoto*): a configured 0 falls back to the built-in
    default instead of rendering the single-click name label black; invis/hidden
    hues keep an explicit 0. sphere.ini docs corrected (hue is hex; 0 = default) and
    the duplicate/incorrectly-"unimplemented" block removed.

LOGGING:
  - Demoted [move_z_delta] (per-step) and [tick_stats] / [gc_stats] (every 30s) from
    INFO to DEBUG so a debug-off console stays quiet; they reappear when packet or
    script debug raises the level. [slow_tick] stays a warning.

[2026-06-06] - Panel update service, smelt/spell item triggers, configurable hues (Wave 72)
----------------------------------------

HOST / PANEL:
  - Added UpdateService and the POST /api/update/run + GET /api/update/status
    endpoints, surfaced as a "System Update / Update from GitHub" card on the
    Settings page (state and last log lines shown live). The update runs git
    status, git pull --ff-only, server save, server stop, build.ps1 -ServerOnly,
    then server start, in that order.
  - build.ps1 gained a -ServerOnly switch that publishes only SphereNet.Server and
    copies the panel static files, leaving the running (and therefore locked)
    SphereNet.Host.exe untouched so the panel can update itself.

TRIGGERS:
  - @Smelt: double-clicking ore now prompts for a forge target and fires @Smelt
    (N1 = ore amount, O1 = forge, S1 = smelt message) before the native smelt;
    RETURN 1 cancels it. On success the ore stack is consumed and an iron ingot
    stack is added to the backpack (or dropped if there is no pack), firing
    @Create on the result.
  - Item-targeted @SpellEffect: Mark, Recall/Gate rune travel, Dispel Field and
    generic TargObj item targets now fire @SpellEffect (O1 = caster, N1 = spell id,
    S1 = spell name); RETURN 1 suppresses the native behaviour. SpellEngine takes
    a TriggerDispatcher, wired from Program.

CONFIG:
  - Notoriety and invisibility hues are configurable via sphere.ini: ColorNotoGood,
    ColorNotoGoodNPC, ColorNotoGuildSame, ColorNotoNeutral, ColorNotoCriminal,
    ColorNotoGuildWar, ColorNotoEvil, ColorNotoInvul, ColorNotoInvulGameMaster,
    ColorNotoDefault, ColorInvisItem, ColorInvis, ColorInvisSpell, ColorHidden.
    NotoToHue now resolves through the loaded NotorietyHueSettings, distinguishing
    NPC innocents (GoodNpc) and GM-level invulnerables (InvulGameMaster). Hue keys
    parse hex; PetsInheritNotoriety and AdvancedLos keys are also read. Hex/decimal
    parsing handled by GetHue / GetIntOrHex helpers.

NPC AI:
  - Consolidated every NPC move/wander/chase/path-step gate into a single
    CanNpcMoveTo check: negative-coordinate guard, tile passability, static-block
    items and occupying characters, all skipped when the NPC can pass walls
    (hidden/invisible staff above the occupied tile do not block).

NETWORK:
  - 0xBE now distinguishes the modern length-prefixed assist/version packet from
    the legacy 5-byte uint form, via TryGetLegacyAssistPacketLength: implausible
    declared lengths followed by a real packet boundary are treated as the legacy
    form so following movement packets are not stranded behind a phantom partial.
  - 0xD9 Hardware Info length corrected to 0x010C (268 bytes total).

SCRIPTING:
  - SRC-aware dialog verb dispatch: UID.<obj>.DIALOG and similar verbs now route
    through _REF_EXEC_AS (srcUid|targetUid|command|args) when SRC is known, so
    client-bound verbs open on SRC's client while the UID object stays the dialog
    subject. HandleDialogCommand / OpenNamedDialog thread the subject through.
  - TriggerRunner stops a trigger body at any following ON= key (not only @-named
    triggers), so ON=*keyword* speech blocks no longer bleed sibling responses
    into one spoken line.

TRIGGERS:

----------------------------------------

TRIGGERS:
  - Added per-NPC seen-player memory and wired @NPCSeeNewPlayer.

[2026-06-06] - Client speed mode (0xBF sub 0x26) parity (Wave 58)
----------------------------------------

PROTOCOL / PACKET:
  - Added 0xBF sub 0x26 (PacketSpeedMode) outgoing packet, carrying the client
    movement speed mode byte.

MOVEMENT:
  - SPEEDMODE bit 0 makes a foot character move at the mounted cadence
    (walk/run 200/100ms instead of 400/200ms), matching Source-X Event_Walk.
    Added a speedMode-aware GetMoveDelay overload; the per-step move delay now
    derives from the character's SpeedMode.
  - The speed-hack detector treats a speed-mode character as mounted so the
    faster cadence is not flagged as a violation or counted as a burst.
  - SendSpeedMode pushes the current mode to the active client and resets the
    walk validator (token/throttle state) so the new cadence applies cleanly.

GM / COMMANDS:
  - SPEEDMODE is a GM command (".speedmode <n>"); it sets the property and echoes
    the resulting value.
  - The mode is (re)sent to the owning client on login and whenever SpeedMode
    changes via a command or a .set property edit, so client and server agree on
    cadence without a relog.

[2026-06-05] - Combat attack handshake parity and NPC special-attack facing (Wave 57)
----------------------------------------

PROTOCOL / PACKET:
  - Added 0x2F (combat swing/fight notification: attacker + defender serials)
    and 0xAA (attack target confirmation; serial 0 means refused) outgoing
    packets.

COMBAT:
  - Attack requests now reply with an accept/refuse handshake. A refused attack
    (target 0/self, a hidden staff target, region/PvP block, or an Attack /
    CombatStart trigger returning true) sends 0xAA=0; an accepted attack sends
    0x2F Swing followed by 0xAA carrying the target serial.
  - War mode is entered and the fight target is set only on an accepted attack.
    Previously the request forced war mode (and assigned FightTarget) up front,
    before the rejection checks, so a refused attack still flipped the attacker
    into war mode and left a stale target.

NPC AI:
  - Breath and throw special attacks turn the NPC to face the target and
    broadcast a 0x77 direction update before emitting the effect, so the
    projectile/breath originates from the correct facing instead of the NPC's
    previous direction (FaceAndBroadcastToward).

[2026-06-05] - Turn-in-place movement parity, walk-token config fix, reject diagnostics (Wave 56)
----------------------------------------

MOVEMENT:
  - Turn-in-place parity: a 0x02 walk request whose direction differs from the
    character's current facing is now handled as a pure turn, not a tile step.
    It only updates facing, sends 0x22 MoveAck, and broadcasts 0x77, bypassing
    the throttle, walk-token bucket, collision and Z resolution. Previously such
    a packet advanced the character one tile, so the client's direction-change
    "doublet" packets (~31-47ms apart while running) produced an extra server
    step, a momentary animation hitch, and a sideways snap near stairs/walls.
  - Walk-token bucket is honored as documented and no longer starves movement on
    a low setting: WalkBuffer is the bucket capacity (allowed step burst) and
    WalkRegen is tokens-per-SECOND (a rate, not a per-step tick period). A
    running mover needs ~5 steps/s, so a regen below that drained the bucket and
    rejected legitimate steps as walk_buffer, snapping the client back into a
    repeated stutter. Config restored to working values (75 / 25) and the stale
    "loaded but unused" comments corrected. Validate() now warns when WalkRegen
    or WalkBuffer is set below running speed.
  - Post-reject in-flight steps (seq > 1 arriving after a reject reset
    WalkSequence to 0) are dropped silently — no extra 0x21 reject and no 0x20
    DrawPlayer redraw. Echoing either made the client ClearSteps and snap again,
    cascading into a client resync (the visible walking "teleport"/flash). The
    fixed post-reject absorb window (MoveRejectResyncMs) is disabled accordingly.
  - GM/staff movement is routed through the movement queue instead of an
    immediate step, so the client's transition doublet is paced to the natural
    cadence; the throttle is still skipped for staff, so GM is smoothed but never
    throttle-rejected. Movement speed is unchanged.
  - Queued movement is drained every main-loop iteration (not only on the 50ms

[2026-06-01] - Non-blocking (async) game send (Wave 46)
----------------------------------------

NETWORK:
  - The game stream no longer uses a blocking Socket.Send. A slow client, a full
    OS send buffer, or a packet burst could previously block the flush thread
    that was serving it. The game socket is now set non-blocking on login, and
    FlushOutput accumulates compressed+encrypted bytes into a persistent
    per-connection buffer ([_outStart.._outEnd), stream-ordered) and drains only
    what the socket accepts (Send with out SocketError; WouldBlock leaves the
    remainder for the next flush). The server thread never blocks.
  - Backpressure: a connection buffering more than 512 KB of unsent bytes is
    hopelessly behind and is disconnected (bounds worst-case memory to
    ~maxClients * 512 KB). Clear() now does a best-effort final flush before
    shutdown so a graceful disconnect still delivers queued/pending data.
  - Design note: a synchronous non-blocking + per-tick retry model is used rather
    than SocketAsyncEventArgs/SendAsync — it reaches the same goal (no blocked
    thread) without completion-callback races, in-flight tracking and post-
    completion cleanup, which are the parts most likely to produce wire bugs
    worse than blocking. One active send per connection is inherent (flush is
    serial per connection), packet order is preserved (single ordered buffer),
    and the shared-broadcast/pre-compress path is unchanged. Toggle:
    NetState.NonBlockingGameSend (default on) keeps the old blocking path for A/B
    and rollback.

  Validated: 529 tests; 300-combat correctness run (zero wire corruption, zero
  backpressure trips — healthy connections drain fully each tick, tick
  unchanged); 1,000-bot cluster — zero corruption and zero final-flush errors
  across ~1,900 teardowns, the server never blocks, slow clients are shed cleanly
  by the byte cap instead of a blocking cascade, peak RSS ~2.1 GB (was ~4.6 GB at
  a 2 MB cap). The in-process bot cluster's eventual attrition is unchanged — it
  is the bots being CPU-starved, not the send path — but the failure mode is now
  bounded and non-blocking.

[2026-05-31] - Pool outgoing packet buffers (Wave 39)
----------------------------------------

PERFORMANCE / NETWORK:
  - Every outgoing packet allocated a fresh byte[] backing array (PacketBuffer),
    the bulk of a packet's allocation. PacketBuffer now rents its backing array
    from ArrayPool<byte>.Shared and returns it once FlushOutput has consumed the
    bytes. This is safe because every outgoing PacketBuffer is built, queued and

    scales with outbound packet volume (dense clusters / large battles). Verified
    correct under a sustained 300-bot run: 300/300 connected, 0 disconnects, no
