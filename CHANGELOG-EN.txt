[2026-07-02] - Parity wave W-H6: custom house fixtures, EQUIP* verbs, client CAST
----------------------------------------

ADDED - Custom-house commit materializes real fixtures (Source-X CommitChanges):
  - INTERACTIVE design tiles — doors and containers — become real items on
    commit (they open, close and hold contents); walls/floors stay virtual.
    Door fixtures link to the multi (the house key opens them), get
    Move_Never and register as house components. A recommit tears down the
    previous fixture set (COMMIT_FIXTURES tracking). Detection uses ITEMDEF
    TYPE first, then the tiledata Door/Container flags.

ADDED - Char verbs EQUIPHALO / EQUIPARMOR / EQUIPWEAPON:
  - EQUIPHALO equips a glowing light source (arg = lifetime seconds);
    EQUIPARMOR dresses pack armor into free layers; EQUIPWEAPON equips the
    highest-attack weapon when the hands are free.

ADDED - Client verbs OPENPAPERDOLL and CAST:
  - OPENPAPERDOLL opens the target's (or own) paperdoll; CAST starts a spell
    from script by number or defname (targeted spells open the cursor).

[2026-07-02] - Parity wave W-H5: NPC skill training and ground-food seeking
----------------------------------------

ADDED - NPC skill training (Source-X NPC_OnTrainHear/Pay):
  - "train" makes a townsfolk NPC list the skills it can teach (mastered,
    >= 10.0); "train <skill>" quotes a price (exact or prefix name match).
    A trainer teaches up to 30% of its own skill, hard-capped at 42.0
    (Source-X TRAINSKILLPERCENT/TRAINSKILLMAX defaults).
  - Gold handed to the trainer while the offer is open buys skill: 1 gp =
    0.1 skill; overpayment beyond the cap bounces back to the student and
    reaching the cap closes the offer.

ADDED - NPC ground-food seeking (Source-X NPC_Act_Food):
  - Hungry creatures no longer eat only from their packs: within a range
    that widens with hunger (sight * (100-food)%), they find edibles on the
    ground, walk toward them and eat when adjacent. Grazing is preserved.

----------------------------------------

The first seven waves of the category-by-category Source-X audit
(wiki/hedef.txt) were applied in one pass. Test suite grew 1040 -> 1076
(all green, ParityWaveA-G guardrail suites). Highlights:

FIXED - Trigger contracts:
  - Trigger names aligned with the Source-X tables: @Jailed, @Ship_Move/
    _Stop/_Turn, @itemSPELL (scripts using the reference names never fired).
  - @Hit/@HitTry/@HitCheck/@HitMiss: SRC=victim, ARGO=weapon; damage-type
    ARGN2 on @Hit/@GetHit/@HitCheck; @HitCheck ARGN1=swing state.
  - @Death SRC=the dying char (killer on ARGO), @Kill ARGN1=attacker count,
    item @Step ARGN1=standing flag.
  - @SpellEffect fires on the AFFECTED char (SRC=caster, ARGN2=skill level,
    writable LOCAL.Effect/Resist/Duration, RETURN 1 cancels per target);
    @SpellCast seeds ARGN2=difficulty + writable ARGN3 cast time.
  - [SKILL n] section stages (ON=@Start/@Stroke/...) execute for the first time.

FIXED - Combat core:
  - Players now gain weapon skill + Tactics from combat (never gained before).
  - Era-0 hit roll is the two-stage model: random difficulty 0..iDiff checked
    against the weapon-skill bell curve; frozen targets are near-certain hits.
  - Coverage-weighted whole-body AR replaces the single-region reduction;
    legacy parry formula (shield/40, DEX<80 erosion, GM +5, trains Parrying).
  - IT_TRAP dclick SPRINGS the trap (full Use_Trap state machine); equipped
    cursed items refuse to move; paralyze breaks on damage; Dispel strips
    active buffs/curses.

FIXED - Notoriety follows the Source-X decision order:
  - Party/guild resolve BEFORE criminal/murderer (a party-member murderer
    renders green); TAG.OVERRIDE.NOTO honored; PLAYEREVIL/PLAYERNEUTRAL
    karma thresholds added (sphere.ini); murderers see themselves red; your
    own pet renders neutral (the Source-X default).

ADDED - NPC AI / economy / social:
  - NPC hostility: karma alignment + CREID ally-group families (orcs no
    longer attack orc mages) + fight-memory grudges.
  - The vendor purse is always tracked (restock funds it, partial sell fill
    with a shortfall bark); VENDORMARKUP drives the sell payout.
  - Guild stone gump: dismiss member, declare/break alliance, set name/
    abbreviation, fealty vote. Party member health bars update beyond
    visual range.
  - Region names are announced only in ANNOUNCE-flagged regions.

ADDED - Housing/ships/persistence:
  - House placement generates a pack key + bank spare; lockdown/secure stamp
    the item ATTR; redeed sweeps loose floor items into the moving crate.
  - Ship planks open/close (5s autoclose); ships collide with statics and
    other hulls.
  - Sector environment (light/season/weather), ALL memory types (with
    remaining timeouts) and the attacker log survive save-load.
  - SERV.B / SERV.GARBAGE / SERV.INFORMATION bridged to scripts; DISTANCE,
    HITPOINTS/USESCUR aliases, item CONSUME/BOUNCE/DECAY verbs.

[2026-07-01] - Network: server-initiated RTT ping disabled by default (Wave 210)
----------------------------------------

FIXED - The client ping meter showed absurd values (hundreds of thousands of ms):
  - The server sent an unsolicited 0x73 ping to each client every 30s to measure its
    own RTT. The stock ClassicUO client feeds EVERY incoming 0x73 into its own ping
    average (Time.Ticks minus a baseline that only advances when the client itself sends
    a ping), so the server's pings made the client display its own uptime as "ping" —
    an average in the 500k ms range with spikes to millions, dropping to the real 8-12ms
    only in the instant after the client sent its own ping. The server-side RttMs value
    was never consumed anywhere. The feature now ships disabled (RttPingIntervalMs=0);
    the client's own ping still works (the server echoes client-initiated pings). Set
    RttPingIntervalMs to a positive value only with a client that ignores unsolicited
    pings. NOTE: if RttPingIntervalMs is set in your .ini, change it to 0 there too.

[2026-07-01] - Persistence parity: active poison survives save-load (Wave 209)
----------------------------------------

FIXED - An active poison was lost on save-load:
  - A poisoned character's level, remaining ticks, next-tick schedule and poisoner were
    never written to the world save, so the poison silently ended on restart (and a
    poison kill lost its attribution). The poison is now saved as remaining time + tick
    count (like item decay / TIMERF) and restored exactly on load via a new
    CharacterPoisonState.Restore, so it resumes rather than re-arming fresh.

KNOWN - Active spell buffs/debuffs still become permanent across a restart:
  - Stat effects (Bless/Curse/Strength/Weaken/...) modify the character stat directly
    and are runtime-only, so save-load keeps the buffed stat but drops the expiration —
    the buff never wears off after a restart. Fixing this needs a design decision
    (persist the effect list so it expires, or revert effects at save so base stats are
    stored); tracked in PARITY_MATRIX.md rather than rushed here.

[2026-07-01] - Persistence parity: TIMERF timers survive save-load (Wave 208)
----------------------------------------

FIXED - A pending TIMERF / TIMERFMS timer was lost on save-load:
  - Per-object delayed function/verb timers (AddTimerF) were never written to the world
    save, so a scheduled delayed action was dropped when the server restarted — unlike
    Source-X, which keeps object timers across a save. Each pending entry is now saved
    per object as remaining time (not the absolute tick, which resets on restart) and
    re-scheduled on load, mirroring how item decay/TIMERMS already persist. The saver
    emits one TIMERF property per entry (remainingMs|function|args); the loader routes
    it through TrySetProperty, which re-adds the timer with the remaining delay.

[2026-07-01] - Scripting parity: RETURN preserves string values (Wave 205)
----------------------------------------

FIXED - RETURN coerced string values to a number:
  - Both RETURN handlers evaluated the argument as a long and stored that number, so a
    [FUNCTION] returning a name, defname or message (RETURN of a resolved string)
    collapsed to a digit instead of its text. The expression evaluator only returned a
    long with no way to signal "this was not numeric", so the two cases were
    indistinguishable.
  - Added ExpressionParser.TryEvaluate, which reports whether the whole expression was a
    genuine numeric value — it clears an internal flag when an atom resolves to a
    non-numeric string (an unresolved bareword, or a <...> that expanded to text) or
    when content is left unparsed. RETURN now keeps the string when the arg is not
    numeric, and still evaluates numbers, hex and expressions. Plain Evaluate is
    untouched, so no numeric path changes.

[2026-07-01] - Trigger parity: ARGS string mutation (Wave 204)
----------------------------------------

FIXED - Scripts could not rewrite a trigger's string argument (ARGS):
  - <ARGS> was seeded and readable, but a line like ARGS=<new text> fell through to a
    no-op (the interpreter had no ARGS assignment path) and RunWrapped never copied the
    string back, so trigger text was read-only. ARGS assignment now writes the trigger
    arg and the rewrite is copied back to the caller — the mechanism a script needs to
    reshape trigger text (Source-X @Speech-style rewrite). FireResourceTrigger and
    FireSpellTrigger, which duplicated the ARGN copy-back inline, now share RunWrapped
    so they pick up the ARGS write-back too.

[2026-07-01] - Trigger parity: ARGN write-back from every trigger source (Wave 203)
----------------------------------------

FIXED - ARGN mutations were dropped from most trigger sources:
  - Only object EVENTS lists and character def triggers ran through RunWrapped, the
    method that copies a script's ARGN1/2/3 changes back into the caller's args. Item
    ITEMDEF / TEVENTS / TYPEDEF triggers (where most item behaviour lives), region and
    room events, and all global (EVENTSPLAYER / EVENTSPET / EVENTSITEM / EVENTSREGION)
    events called the runner directly and discarded the mutations — so an item @DClick
    that set ARGN1, or a global event adjusting a trigger number, had no effect on the
    caller. Every trigger firing path now goes through RunWrapped, so ARGN write-back is
    uniform regardless of where the trigger is defined.

[2026-07-01] - Trigger parity: ARGN3 seeding and ARGN mutation (Wave 202)
----------------------------------------

FIXED - <ARGN3> read 0 inside every trigger:
  - TriggerDispatcher.WrapArgs built the script args with only ARGN1/ARGN2 (the
    constructor's two number slots) and never seeded Number3, so a trigger fired with
    N3 set — e.g. @DropOn_* passing the drop Z, @SkillUseQuick passing the result —
    saw <ARGN3> as 0. WrapArgs now seeds Number3 as well.

FIXED - Scripts could not modify ARGN1/ARGN2/ARGN3:
  - The interpreter had no assignment path for ARGN, so a line like ARGN3=99 fell
    through to a no-op property-set — trigger numbers were effectively read-only,
    defeating the RunWrapped copy-back for script-driven changes (Source-X @Hit damage,
    @DropOn drop-Z, @NPCActFight dist/motivation). ARGN1/ARGN2/ARGN3 (and bare ARGN =
    ARGN1) assignment now writes the trigger args, and the value is copied back to the
    caller after the block.

[2026-07-01] - Scripting parity: TRYSRV runs at server privilege (Wave 201)
----------------------------------------

FIXED - TRYSRV did not elevate to server privilege:
  - TRYSRV is meant to run a verb as the server (Source-X g_Serv, PLEVEL 7) so it
    bypasses the calling source's privilege restriction, but it passed the original
    source through — a low-plevel player, or a Guest-level null console — so a
    privileged verb under TRYSRV still failed the very check it exists to bypass. It
    now runs the verb under a server (Owner) source; the original source's script
    bridge stays as the fallback so client-scoped verbs still reach a client.

ADDED - TRY-family coverage: tests pin TRYSRV's server elevation and TRYP's
  privilege gate (verb blocked below the required PrivLevel, allowed at or above).

[2026-07-01] - Scripting parity: script <LINK> reads the object's own link (Wave 199)
----------------------------------------

FIXED - <LINK> in scripts was aliased to the ACT object instead of the object's link:
  - The interpreter resolved both <ACT> and <LINK> to the trigger's second object
    (Object2), so <LINK> never reflected an object's actual m_uidLink — e.g. a lever's
    linked door was unreachable from <LINK> / <LINK.NAME>. <LINK> (and <LINK.prop>) now
    delegate to the current object's LINK property, which already resolves the linked
    object and its sub-properties for both items and characters. <ACT> still resolves
    Object2, so the two references are now independent (VarObjs chain consistency).

[2026-06-30] - Scripting parity: TIMERFMS and delayed-verb timers (Wave 198)
----------------------------------------

ADDED - TIMERFMS object verb (Source-X millisecond timer):
  - TIMERFMS <ms>, <payload> schedules a delayed function/verb on an object in
    milliseconds, the counterpart of TIMERF's seconds (which was the only unit
    supported). Both now share one parser (ObjBase.ScheduleTimerF) that differs only

  - The timer executor only tried the payload as a script function, so a bare verb
    such as TIMERF 5, SAY hi or TIMERF 10, REMOVE never ran. It now falls back to
    executing the payload as a command on the object when no matching function
    exists, matching Source-X TIMERF (delayed verb OR function).

[2026-06-30] - Vendor parity: refuse to buy a non-empty container or immovable item (Wave 195)
----------------------------------------

FIXED - Selling a non-empty container to a vendor destroyed its contents:
  - VendorEngine.ProcessSell validated amount, gold and the optional buy-list filter
    but not the item itself, so a vendor with no buy list (buys anything) would accept
    a bag and Delete() it — silently eating every item inside for the bag's sell price
    (an item-loss exploit). The sell path now refuses an immovable item (Move_Never)
    and any container that still holds items, mirroring ServUO BaseVendor's sell guard
    (skip !Movable and a Container with Items.Count != 0). An empty container stays
    sellable.

    in the wearer's equipment slot — dead references that inflated object counts
    and could surface as a deleted item still counting toward equipment/weight.
    These paths now route through the full world removal, which unlinks the item
    from the object table, parent container, equipment slot and sector. Object
    teardown also clears the wearer's layer when an equipped item is removed.

FIXED - World / Regions:
  - FindRegion cached lookups per 8x8 tile cell but also stored null (no-region)
    results. Region edges are not 8-aligned, so a single cell can straddle a
    boundary; a region-less query then poisoned the cell and a later in-region
    query for another tile in the same cell wrongly returned null. This could let
    region-gated behavior (guarded, no-PvP, no-magic, enter/leave, gathering,
    weather) lapse near boundaries. Only successful lookups are cached now;
    non-region tiles re-resolve, and cached hits stay self-validating.

FIXED - Items / Inventory:
  - Merging a dragged stack onto a target stack that sits on the ground sent a
    container-content packet addressed to the ground item itself, desyncing its
    amount label. The merged ground pile now broadcasts a world item update, and
    any overflow remainder drops to the ground beside it instead of being lost.

FIXED - Host / Stability:
  - The Host had no last-resort handler for unhandled exceptions on background
    threads. The child server's output is mirrored to the console on an
    AsyncStreamReader thread with no surrounding try/catch, so when the console
    handle became invalid mid-run (an interactive/RDP session disconnecting, or
    redirected stdout) the resulting IOException went unhandled and terminated
    the Host, taking the child game server down with it and leaving only a
    0xe0434352 in the event log. Console mirroring now swallows IOException and
    InvalidOperationException, and a global handler persists any unhandled
    exception to logs/host-crash.log.

FIXED - Memory:
  - Server and Host now build with Workstation GC instead of Server GC. On the
    low-core/low-RAM VDS deployment target Server GC keeps a per-CPU heap and
    holds committed segments, so RSS climbed steadily under a steady allocation
    rate and read as a leak even with a single player.
  - NetState releases its grown per-connection send and crypt scratch buffers
    back to their initial sizes when a connection is cleared, so a transient
    large broadcast no longer pins that memory for the life of the slot.

SECURITY:
  - Pinned the native SQLite binary (SQLitePCLRaw.lib.e_sqlite3 3.50.3) above
    the 2.1.11 pulled in transitively by Microsoft.Data.Sqlite, clearing
    advisory GHSA-2m69-gcr7-jv3q (NU1903). The package is native-only, so the
    managed SQLitePCLRaw core/provider ABI is unchanged.

[2026-06-05] - Turn-in-place movement parity, walk-token config fix, reject diagnostics (Wave 56)
----------------------------------------

MOVEMENT:
  - Turn-in-place parity: a 0x02 walk request whose direction differs from the
    character's current facing is now handled as a pure turn, not a tile step.
    It only updates facing, sends 0x22 MoveAck, and broadcasts 0x77, bypassing
    the throttle, walk-token bucket, collision and Z resolution. Previously such
    a packet advanced the character one tile, so the client's direction-change
    "doublet" packets (~31-47ms apart while running) produced an extra server
    step, a momentary animation hitch, and a sideways snap near stairs/walls.
  - Walk-token bucket is honored as documented and no longer starves movement on
    a low setting: WalkBuffer is the bucket capacity (allowed step burst) and
    WalkRegen is tokens-per-SECOND (a rate, not a per-step tick period). A
    running mover needs ~5 steps/s, so a regen below that drained the bucket and
    rejected legitimate steps as walk_buffer, snapping the client back into a
    repeated stutter. Config restored to working values (75 / 25) and the stale
    "loaded but unused" comments corrected. Validate() now warns when WalkRegen
    or WalkBuffer is set below running speed.
  - Post-reject in-flight steps (seq > 1 arriving after a reject reset
    WalkSequence to 0) are dropped silently — no extra 0x21 reject and no 0x20
    DrawPlayer redraw. Echoing either made the client ClearSteps and snap again,
    cascading into a client resync (the visible walking "teleport"/flash). The
    fixed post-reject absorb window (MoveRejectResyncMs) is disabled accordingly.
  - GM/staff movement is routed through the movement queue instead of an
    immediate step, so the client's transition doublet is paced to the natural
    cadence; the throttle is still skipped for staff, so GM is smoothed but never
    throttle-rejected. Movement speed is unchanged.
  - Queued movement is drained every main-loop iteration (not only on the 50ms
    tick), so the ack/reject flushes in the same iteration as input. The per-step
    _nextMoveTime gate still paces the real drain, so this only cuts latency for
    the client's 5-step budget on fast/stair sections; it does not raise speed.

DIAGNOSTICS:
  - Movement rejections are classified into a normalized cause (throttle,
    walk_buffer, descent_cap, impassable_static, no_surface, z_window,
    diagonal_edge, mob_block, seq_mismatch, queue_full, stale) and accumulated
    into a per-session [move_reject_tally] summary, so the dominant disagreement
    class over a walking session can be aimed instead of guessed. Pure logging.

[2026-06-01] - Non-blocking (async) game send (Wave 46)
----------------------------------------

NETWORK:
  - The game stream no longer uses a blocking Socket.Send. A slow client, a full
    OS send buffer, or a packet burst could previously block the flush thread
    that was serving it. The game socket is now set non-blocking on login, and
    FlushOutput accumulates compressed+encrypted bytes into a persistent
    per-connection buffer ([_outStart.._outEnd), stream-ordered) and drains only
    what the socket accepts (Send with out SocketError; WouldBlock leaves the
    remainder for the next flush). The server thread never blocks.
  - Backpressure: a connection buffering more than 512 KB of unsent bytes is
    hopelessly behind and is disconnected (bounds worst-case memory to
    ~maxClients * 512 KB). Clear() now does a best-effort final flush before
    shutdown so a graceful disconnect still delivers queued/pending data.
  - Design note: a synchronous non-blocking + per-tick retry model is used rather
    than SocketAsyncEventArgs/SendAsync — it reaches the same goal (no blocked
    thread) without completion-callback races, in-flight tracking and post-
    completion cleanup, which are the parts most likely to produce wire bugs
    worse than blocking. One active send per connection is inherent (flush is
    serial per connection), packet order is preserved (single ordered buffer),
    and the shared-broadcast/pre-compress path is unchanged. Toggle:
    NetState.NonBlockingGameSend (default on) keeps the old blocking path for A/B
    and rollback.

  Validated: 529 tests; 300-combat correctness run (zero wire corruption, zero
  backpressure trips — healthy connections drain fully each tick, tick
  unchanged); 1,000-bot cluster — zero corruption and zero final-flush errors
  across ~1,900 teardowns, the server never blocks, slow clients are shed cleanly
  by the byte cap instead of a blocking cascade, peak RSS ~2.1 GB (was ~4.6 GB at
  a 2 MB cap). The in-process bot cluster's eventual attrition is unchanged — it
  is the bots being CPU-starved, not the send path — but the failure mode is now
  bounded and non-blocking.

[2026-05-31] - Shared broadcast packets — raise the dense-crowd wall (Wave 42)
----------------------------------------

[2026-05-31] - Pool outgoing packet buffers (Wave 39)
----------------------------------------

PERFORMANCE / NETWORK:
  - Every outgoing packet allocated a fresh byte[] backing array (PacketBuffer),
    the bulk of a packet's allocation. PacketBuffer now rents its backing array
    from ArrayPool<byte>.Shared and returns it once FlushOutput has consumed the
    bytes. This is safe because every outgoing PacketBuffer is built, queued and
    flushed exactly once — broadcasts build a fresh packet per recipient (they
    pass a PacketWriter, not a shared buffer), so no buffer is ever shared across
    send queues and no reference counting is needed. Buffers are also returned on
    the queue-overflow, not-in-use and disconnect paths so they never leak from
    the pool. Incoming/wrapped buffers (the byte[] constructor) are never pooled.
  - Growth (EnsureCapacity) stays within the pool: rent a larger array, copy,
    return the old one. Rent may hand back an oversized array; all readers use
    the logical Length (never _data.Length), so the slack is never observed.
  - Removes the per-packet byte[] allocation from the send hot path; the benefit
    scales with outbound packet volume (dense clusters / large battles). Verified
    correct under a sustained 300-bot run: 300/300 connected, 0 disconnects, no
    malformed/underrun packets — pooling does not corrupt the wire.

  - A* path recompute is now throttled per NPC (at most once per ~750ms). In a
    churning crowd the cached next step is invalidated almost every tick (a
    neighbour moves onto it), which previously forced a fresh A* search every
    tick. Between allowed recomputes the NPC follows its cached path or faces
    the target and holds. A dedicated throttle timestamp survives path-cache
    drops (and is swept for gone NPCs) so the limit actually binds.
  - Helps real crowded combat (e.g. a champion spawn where mobs jostle); the
    pathological 50k-clustered stress remains dominated by other per-NPC
    combat-brain cost that needs profiling to isolate further.

[2026-05-31] - NPC target-scan bounding in same-faction crowds (Wave 31)
----------------------------------------

PERFORMANCE:
  - FindBestTarget now caps the number of attackable candidates it evaluates
    (not just hostile ones). Wave 30's cap only counted positive-motivation
    targets, so a monster surrounded by its own kind still evaluated every
    neighbour before finding the few real enemies — O(local density) per NPC.
    It now considers its nearest handful; a target farther back in a crowd is a
    closer NPC's problem, so combat still engages.
  - GetAttackMotivation does the cheap hostility test before the FindRegion
    lookup, so non-hostile candidates (the bulk of a crowd) skip the region
    query.
  - Note: the residual cost of thousands of monsters densely clustered around
    players is pathfinding/movement in the crowd, which is inherent to that
    (artificial) density and unaffected here; normal play is unaffected.

[2026-05-31] - NPC AI tick-cost bounding under crowded combat (Wave 30)
----------------------------------------

PERFORMANCE:
  - Per-tick NPC budget: a mass sector activation (a player entering a densely
    populated area) could make thousands of NPCs due in one tick and run their
    AI serially, stalling the tick for several seconds. The tick now processes
    at most a bounded number of NPCs and defers the overflow to the next tick.
  - Bounded target acquisition: FindBestTarget previously did a line-of-sight
    raycast (against the real static map — the dominant cost) for every nearby
    mobile, making acquisition O(crowd) per NPC. It now ranks candidates by
    motivation without LOS, caps how many are evaluated, and verifies LOS lazily
    only on the top few — a few raycasts per NPC regardless of crowd size.
  - Net: a pathological stress (50,000 monsters clustered around 8 fighting
    players) dropped from ~8.8s ticks to ~2.2s; normal and realistic loads are
    unaffected. (525 tests pass.)
